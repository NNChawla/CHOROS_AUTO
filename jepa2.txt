# Critical issues to fix first

## 1. `mean_std` pooling will break downstream dimensionality unless the embedding pipeline handles `2D`

The model’s `mean_std` pooling returns `(B, 2D)` instead of `(B, D)`.  That is valid mathematically, but your checkpoint stores `embed_dim`, not the actual output dimension. If `embed_target_data.py` or downstream probes assume embeddings are always `embed_dim`, `mean_std` will cause shape mismatches or silent schema inconsistency.

Fix by storing:

```python
'embedding_out_dim': args.embed_dim * 2 if args.embed_pool == 'mean_std' else args.embed_dim
```

And make sure the embedding pipeline reads it.

For the first pilot, I would use:

```bash
--embed_pool mean
```

not `mean_std`.

---

## 2. Future-target sampling can fail for short valid sequences

In `_sample_masks()`, future mode does:

```python
H = random.randint(lo, min(hi, K, nv - future_min_gap - 1))
```

If `nv - future_min_gap - 1 < lo`, this can produce an invalid `randint` range. 

You need a guard before sampling future targets:

```python
max_H = min(hi, K, nv - future_min_gap - 1)
if max_H >= lo:
    H = random.randint(lo, max_H)
    ...
else:
    mode = "masked_span"
```

Right now, `mixed` mode may randomly choose `future` and crash on short sequences. This is probably the most important correctness fix.

---

## 3. Patch validity currently drops partial patches

The code computes valid patches as:

```python
n_valid_patches = lengths // P
```

and patchifies only `x[:, :N * P]`.  This means any partial final patch is treated as invalid even if it contains valid frames.

That is acceptable if all sequences are padded/cropped to fixed windows and most are full length. But for shorter clips, this discards valid data. If you want partial patches to count, use:

```python
n_valid_patches = torch.ceil(lengths.float() / P).long()
```

Then you also need to ensure padded frames inside partial patches are zeros and not treated as real signal.

For now, this is not fatal, but document it as:

```text
Only full patches are considered valid.
```

---

## 4. The “last” pooling implementation is wrong for padded sequences

Current code:

```python
last = (N - 1 - valid_b.flip(1).long().argmax(1)).clamp(min=0)
```

where `valid_b = ~patch_pad`. 

This finds the position of the first `True` in the reversed valid mask. For a typical valid mask like:

```text
[True, True, True, False, False]
```

the reversed mask is:

```text
[False, False, True, True, True]
```

`argmax` returns index 2, so `last = 4 - 2 = 2`, which is correct. But if a sequence has **no valid patches**, `argmax` returns 0 and `last = N-1`, which is wrong. You may avoid this elsewhere, but inference should be robust.

Safer:

```python
valid_counts = valid_b.long().sum(dim=1).clamp(min=1)
last = valid_counts - 1
```

## 5. `checkpoint_best_probe_val.pt` may save the wrong checkpoint payload

In the training loop, you build `ckpt`, save latest/best, then run periodic eval, then if probe MCC improves, you save the same `ckpt` object as `checkpoint_best_probe_val.pt`. 

But `ckpt['best_probe_mcc']` was created before `best_probe_mcc` is updated. So the saved best-probe checkpoint can contain the old `best_probe_mcc` metadata.

Fix:

```python
if probe_mcc is not None and probe_mcc > best_probe_mcc:
    best_probe_mcc = probe_mcc
    ckpt['best_probe_mcc'] = best_probe_mcc
    torch.save(ckpt, run_dir / 'checkpoint_best_probe_val.pt')
```

The model weights are probably correct for that epoch, but the metadata is stale.

# Important design issues

## 6. `context_group_mask` and `context_device_drop` overlap in training

The training script computes:

```python
_fgm_cols = group_col_indices(...)
feat_mask_cols = ...
ctx_dev_cols = _col_indices_for_context(args)
```

But `_col_indices_for_context(args)` includes both `context_device_drop` and `context_group_mask`.  

Then `jepa_loss()` receives both:

```python
feat_mask_cols=feat_mask_cols,
context_device_cols=ctx_dev_cols
```

So group-mask columns can be included twice. Since zeroing twice is harmless, this is not a numerical bug. But it is confusing and makes the variable names misleading.

I would change `_col_indices_for_context()` to return **only device columns**, or rename the arguments:

```python
context_group_cols
context_device_cols
```

and combine them once inside the model.

---

## 7. The model is now patch-level, but the dataset window length must be divisible by `patch_size`

The model uses:

```python
self.max_patches = max_len // patch_size
N = T // P
```

and silently drops leftover frames.  

That is fine if `max_len % patch_size == 0`. Your defaults `128 / 8` are fine. But the CLI allows arbitrary `--max_len` and `--patch_size`.

Add a hard check:

```python
if max_len % patch_size != 0:
    raise ValueError("max_len must be divisible by patch_size")
```

Also check in the training script before model construction.

---

## 8. Per-sample mask generation uses Python loops and Python `random`

The implementation samples masks per sample using Python loops and `random`.  Conceptually valid, but it will add CPU overhead and is less reproducible under multi-worker/distributed setups than using torch RNG.

For a pilot, this is acceptable. For a large sweep, I would eventually vectorize or at least use `torch.Generator`/torch RNG.

---

## 9. The mixed-mode target sampling does not log the actual fraction of future vs masked targets

`mixed` randomly chooses future or masked-span per sample.  But because future mode can fail and fall back to masked-span, your actual future percentage may be less than 50%.

Add diagnostics:

```text
actual_target_mode_future_frac
actual_target_ratio
actual_valid_target_patches
```

This matters for interpreting runs.

---

## 10. Span target sampling can still underrepresent requested target structure

The code builds a `tgt_set` from multiple spans, then trims or pads to exactly `K`.  This ensures exactly `K` targets, which is good. But trimming with:

```python
tgt_list = tgt_list[:K]
```

biases toward earlier patch indices when the set is too large.

Better:

```python
tgt_list = sorted(random.sample(tgt_list, K))
```

This avoids a systematic early-window bias.

# One likely integration issue to check

Your downstream embedding script needs to know how to instantiate `PoseJEPA`.

The training script saves checkpoints with args including:

```text
patch_size
embed_pool
target_mode
latent_loss
context masks
```



But if `pipeline/embed_target_data.py` only knows how to load `VREncoder` or `TSJEPA`, then periodic eval will fail. The training script calls the existing embed script with only:

```text
--ckpt
--data_dir
--stride
--objective
--target_cols
```



So verify that `embed_target_data.py` can detect a Pose-JEPA checkpoint and instantiate `PoseJEPA` with `patch_size` and `embed_pool`.

This is not a flaw in the model itself, but it is a likely pipeline break.
